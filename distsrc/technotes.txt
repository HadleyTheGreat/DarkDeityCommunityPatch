--------------------------
Dark Deity Community Patch
     TECHNICAL NOTES
--------------------------
This patch is an unofficial bug fix implemented to fix several of the more serious bugs in Dark Deity.  It is my hope that eventually these bugs will be fixed in an official game release, so I am documenting the technical nature of the changes I made and my investigation into them so the nature of bugs and the fixes can be understood by others attempting to refine or implement such bug fixes.

If someone wants to understand more quickly the exact nature of the changes made between the pre-existing code and the changed script code, all you have to do is use the patch installation tool to install the patch then note that when it installs the scripts located in the scripts folder it also extracts the pre-existing scripts into the originals folder, so right after installing the patch you can use a differencing tool like WinMerge to easily compare the new and old code side by side.

I previously discovered that invalid data was making its way in and I made workarounds for these particular crashes in versions 1.0.0 and 1.0.1, but didn't find the root cause of the invalid data.  I've done more in-depth debugging and testing of the game now and located the ultimate source of the data corruption that causes the two bugs that I had previously only made workaroudns for.

I had previously prevented the crashes by just changing the code to detect and ignore invalid data I was finding in the obj_node.viewneighborsranged, obj_node.viewneighborsmelee, and obj_node.neighbors lists, which I was finding to occasionally have objects of types like obj_cursor_follower and various obj_actor descendants instead of just obj_node objects.  This did manage to stop the particular crashes that I knew about, but without resolving the root problem of how invalid data actually got into those lists into the first place so there was no telling if the game might crash in other places that I didn't add a lot of new sanity checks.

To find out what was going on I started out this time with a fresh unmodified version of the game.  I added in a script function to iterate over all of the nodes on the map and check every node to make sure viewneighborsranged and viewneighborsmelee were present, check to make sure they were ds_lists as they should be, and make sure all the entries in them were instances of obj_node that existed (not previously destroyed instances) and pop up a dialog box with show_message immediately to tell me when it detected that the map had become invalid in a way that would cause the crashes that I had previously written sanity checks to work around.  I added calls to that validation function in obj_game's step event so it would happen every time the game draws the screen 60 times a second or whatever it is.  After this I played chapter 22 until it detected that the data was invalid.  If I didn't kill any enemy then I didn't get any invalid map notifications, but as soon as I killed the first enemy I was getting a report that one of the members in the viewneighborsrange lists of one of the nodes on the map was not a valid obj_node object anymore.

This suggested to me that there could be a bug in the actor destroy event that removes the actor from the map when it gets killed, so first I removed my previous calls to the map validation function from obj_game's step event then I went and found the spot in the actor step event obj_actor_Step_0 where it checked to see if an enemy had 0 health and if it did, it called instance_destroy to get rid of that actor on the map.  I put map validation call right before and one right after the instance_destroy call in obj_actor so I could detect whether the problem was occurring inside of instance_destroy, or if it occurred somewhere before the final destruction of the instance.  If was corrupted before entering I would get two duplicate reports upon killing the unit, one going in and one going out of it, or if it was corrupted during the destroy event I would only detect failed validation once on the way out.

I ran the game like this, killed an enemy and got just one report, so the bug was inside of the actor's destroy event, not before it.  I looked at the code in obj_actor's destroy event (obj_actor_Destroy_0) and it was just destroying object instances that it had stored in variables. I rewrote my map validation function so it would take the name of a chackpoint as an argument and if it failed it would pop up a message box with the name of the checkpoint that it had failed at instead of particular information about what type of failure had occurred.  I inserted map validation checkpoints in the obj_actor_Destroy_0 event after it destroyed each of its variable instances to figure out exactly which line was causing the corruption to appear, then ran the game and killed and enemy and it found that the map was fine right up until it freed up the instance of the skillslots.

To look into why freeing skillslots was corrupting the memory I looked at the create event for obj_actor and found that skillslots instance was created with ds_grid_create, so it should be a ds_grid object and the code should call ds_grid_destroy to release it, but what it was actually calling was ds_list_destroy.  Before doing it, it checked ds_exists(skillslots, ds_type_list) to make sure the list existed.  Its perhaps a bug or weakness in this version of GameMaker that ds_exists doesn't return false because it apparently doesn't differentiate between ds_type_list and ds_type_grid and return false if an instance exists, but its not the right type of data structure.  ds_destroy_list also doesn't appear to have any type check to make sure that a type is actually a grid before releasing it.  These two weaknesses led to a situation where the memory somewhere was corrupted because whatever bytes GM changes behind the scenes to mark the variable as released and free the memory for re-use must not be at the same memory offset for ds_list and ds_grid types.  Some of the bytes it changed ended up not being part of the skillslots variable at all, but part of a viewneighborsranged instance also stored in the memory nearby.

Having found this, I made corrections to the destroy event of obj_actor so that instead of checking ds_exists for type ds_type_list and calling ds_list_destroy it would instead check ds_exists for type ds_type_grid and call ds_grid_destroy as it should.  I re-ran the game and tried to kill an enemy and now all my map validations passed and I got no warnings anymore upon killing enemies.  I started the game and tried to kill an enemy and it no longer reported corruption upon killing the first enemy.

This seemed like the kind of mistake that is easy to make and since the version of GameMaker used in Dark Deity didn't seem to be protecting against it I decided to use a grep tool and find check every time in the script code of the game a ds_list was created with ds_list_create, then find every time a ds_list was destroyed with ds_list_destroy and compare the two.  There should always be some corresponding lines in those two lists or one of two problems must exist: either a memory leak exists from not calling ds_list_destroy at all, or a programmer accidentally wrote a call to some other release method like ds_grid_destroy instead ds_list_destroy.  I did the same with ds_grids but they are created in two different ways, one is calling ds_grid_create and the other is calling Read_CSV to read the grid data in from a file.

My findings were that there were several places in the game that both of these bugs were occurring.  The ds_grid/ds_list type mismatch was happening to a few variables in obj_node and one in obj_stat_view, and the failure to release which would lead to a memory leak appeared to be happening in obj_ability_holder, obj_base_bonds3, obj_base_shop_stock, obj_item_holder, and obj_storage_holder.  Most of those failures to release were ds_grid variables created with Read_CSV as opposed to those created by ds_list_create or ds_grid_create.  I made code changes to the destroy events of all of these object types to correct all of these problems.  I have since tested playing all the way through chapter 22 multiple times with only these changes and no other changes I had previously made and I didn't detect any data validation failures and had no crashes, so I am of a strong belief that correcting all of these memory releasing inconsistencies has completely resolved all the original crash bugs in chapter 22.  I didn't re-implement the extensive changes I had made in the earlier patch versions to add sanity checks to several scr_movement scripts and change the way the game was accessing viewneighborsranged, viewneighborsmelee, and neighbors by fixed indexes because those changes are not needed when the game isn't corrupting the data to start with and the less unnecessary changes that are done, the less likely I am to break preexisting code through a typo or an accidental code functionality change.

After this I re-applied changes I had made to obj_turn_scripter_22_Other_10 in version 1.0.1 which was not checking the same square to see if it was empty before spawning a unit on the square due to the check being in GridX, GridY grid coordinates while the spawning was done in x, y pixel coordinates and someone must have made errors when converting between the two or changed one without realizing they had to change the other.

New bug fixes: Movement bugs
-----------------------------
Aside from finding new, more reliable long-term fixes for the bugs I already worked around in the previous patches, I also set out to fix some movement related bugs in the game that I discovered awhile back that for sure occur with the movement speed set to movement speed 8 in the game settings.  To see one easily reproduced example you can move a character exactly 7 squares to the right, then have another character try to push him upwards. It will glitch out visibly. These bugs do not crash the game, but they cause various glitches.

It turns out that if you move a character exactly 7 squares on movement speed 8 then some kind of glitch occurs where the actor does not completely stop moving after it completes its movement path.  Instead of having an x or y value that is a whole number and a multiple of 32, it will have some value after the decimal as a result of drifting very slowly.  If you wait long enough it will drift into another tile which will produce problems like not being able to select the character by clicking on the tile that its sprite appears on, but being able to select it by clicking on another tile which doesn't have its sprite.  This can also result in more than one unit being on the same tile which should not happen.  The x or y value having a decimal portion will make it so that the position of the actor doesn't match the x and y of any obj_node and that will cause malfunction in code that tries to verify that or tries to compare the exact x and y position of the actor with the nodes or with its previous position.  If you try to chain an enemy that is drifting the character will never be able to move again when the chain is removed because it only restores the move speed of the character if it matches the stored x,y position when the chain was created.  If you try to push a character that is not on x,y that have a multiples of 32, you'll see a blurry image where you pushed it from as it is sliding the image until it reaches the exact x,y of the character at which point it will remove the obj_push instance.  That never happens if the actor isn't aligned on x,y that are multiples of 4, which they aren't if the character is drifting.

To work around the several different glitches caused by this drifting problem, I had found in the past that just increasing the movement speed in the options to 10 fixed the specific problem with units moving 7 tiles and didn't appear to create any other new problems, but still since the game defaults to movement speed 8 I believe most players who are unaware will eventually experience these bugs in the game.  I hypothesize that GameMaker should have automatically handled fully stopping the actor but isn't due to some floating point code that adds or subtracts floating point values and expects them to reach some exact amount which does not happen due to normal rounding errors associated with using floating point approximations of real fractions which can have an infinite number of digits after the decimal.  It would happen with some fractions and not with some other fractions so that might explain why it happens at one movement speed but not at another.  I could be totally wrong on that, but it is my assumption.  In any case the solution has to be to manually set the speed of every actor to zero and make sure its x and y are integral values with no digits after the decimal point after it finishes a movement path, so that's what I did. 

The event for finishing a movement path on an actor is obj_actor_Other_8.  This event already exists, so I just modified the code for this event to add lines to set speed, hspeed, and vspeed to 0 in addition to what it was already doing and made sure the grid values it was using to set x and y were whole integers. I tested with this script change applied and it does seem to resolve the drift issues I had previously detected.  Although I didn't test every bug I previously found related to movement speed 8 individually I did spot check to make sure that if I test my easily reproducible case of moving a unit 7 squares to the right, then having another character push it, that does now work properly on movement speed 8.  This means that this fix is correctly detecting and successfully correcting any drift that could have been occurring and it should prevent all the related bugs which only occur when there is drift including any previously undiscovered drifting problems that could possibly have occurred when completing a path with any other arbitrary combination of movement speed and number of squares moved.  A fix like this wouldn't work if there were objects in the game that were intended to keep moving when not on a movement path or which used physics, like in an action arcade game, since it would stop objects that should be moving, but since there are not supposed to be any constantly moving actors in this game except those being driven around by a movement path it should work fine here.

With the resolution of the memory corruption related bugs that led to crashes especially in chapter 22 and the movement drift related bugs that led to some disturbing gameplay glitches on the default movement speed setting, I have fixed all major bugs I know of in the current release version of Dark Deity.  I'm sure those aren't all the bugs in the game but other bugs are much less intrusive on the ability to play the game normally.  I have played all the way through the game in the past and these are the only bugs that I can remember ever giving me headaches, so I hope with version 2.0.0 of the patch everyone can play the game all the way through the game from beginning to end, regardless of their movement speed settings, in a mostly trouble-free manner.

In Version 2.1.0, I fixed a bug where Helena's stat history was not being recorded properly because no line to allocate an obj_stat_holder for her in scr_statadd as it was for all the other characters.

I also fixed an issue with Irving's hair color being the wrong color upon loading a save file from chapter 12-16.  In each of the loading scripts (scr_load_game1, scr_load_game_2, scr_load_game_3, scr_load_game_4, scr_load_game_5, scr_load_game_6, scr_load_game_7, scr_load_game_8, and scr_load_game_battlesave) the game was determining which color Irving's hair should be by checking to see if the scene number was greater than 129, but Irving's hair actually turned black in at the start of chapter 12 which is scene 83 so I corrected the checks to see if the scene was greater than 82.

A reddit user reported to me that he had somehow gotten past round 9 of the chapter 10 battle and the scene did not end.  He said it happened due to loading from a battlesave but I didn't get exact details about how he got his battlesave to level 9 without the map already ending.  I've never experienced or reproduced this bug, but I did make a minor script change so that instead of comparing the turn number to 9 exactly it will attempt to end the map on any turn equal or greater than 9 instead of only exactly equal to 9.  This shouldn't have any consequence for someone without the glitch and it should allow anyone that does manage to run into this problem to finish the map.
