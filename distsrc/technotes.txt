--------------------------
Dark Deity Community Patch
     TECHNICAL NOTES
--------------------------
This patch is an unofficial bug fix implemented to fix several of the more serious bugs in Dark Deity.  It is my hope that eventually these bugs will be fixed in an official game release, so I am documenting the technical nature of the changes I made and my investigation into them so the nature of bugs and the fixes can be understood by others attempting to refine or implement such bug fixes.

If someone wants to understand more quickly the exact nature of the changes made between the pre-existing code and the changed script code, all you have to do is use the patch installation tool to install the patch then note that when it installs the scripts located in the scripts folder it also extracts the pre-existing scripts into the originals folder, so right after installing the patch you can use a differencing tool like WinMerge to easily compare the new and old code side by side.


Version 1.0.0

There are two types of crashes that frequently occur in chapter 22 in the release version of the game which I made workarounds for in version 1.0.0.  I have also seen a bug report on the Steam forum where it appears that crash type 1 happened to someone on Chapter 19 as well.

Crash type 1 - Variable passable not set on an object before reading it
Example crash dump:
___________________________________________
############################################################################################
ERROR in
action number 1
of Step Event0
for object obj_ai_controller:

Variable <unknown_object>.passable(100144, -2147483648) not set before reading it.
at gml_Script_scr_movement_range_ai
############################################################################################
gml_Script_scr_movement_range_ai (line -1)
gml_Object_obj_ai_controller_Step_0


This crash dump indicates that the code tries to access the passable variable but it doesn't exist yet on some object.  Using a grep tool on the decompiled script code dumps it appears to me that the only objects that have the passable variable are obj_node objects.  passable is only checked in two lines of code in all of scr_movement_range_ai  One of those accesses current.passable and the other accesses neighbor.passable.  These are obtained by iterating over the neighbors list with some processing in and out of an open and a closed priority queue to keep track of which nodes have been used and are no longer open for a unit to move to.  It seemed to me like these two variables should be only populated with obj_node objects since those are the only objects that have the passable variable, so I investigated how those get initialized.  Obj_node initializes the passable variable right in obj_node's Create event obj_node_Create_0 so there's no way that any obj_node that was ever created should be missing the passable variable, so my analysis of that is that it must be that the object in current or neighbor was not an obj_node object as expected.

I made a function (scr_debug_assert_instance_type(instance, expected_object_index (type of the object), location (a string that I used pass the name of a checkpoint or the name of a variable to display in the event of an invalid valud detected so I could identify which one of the callers of this function were responsible)).  The function has a var debug defined in its body that if it is true will allow it to pop up a modal dialog window to report the error and if it is false it will not.  Either way the function will have the result true if the assertion passes and all is as expected and false if it does not pass and the object instance is not of the type that was being asserted or the object instance doesn't exist.  I called this function in the two places in scr_movement_range_ai where I replaced an existing check of if (instance_exists({variable})) with if (instance_exists({variable}) && scr_debug_assert_instance_type({variable}, obj_node, "movement_range_ai:{variable}") to prevent the node from being processed if the instance isn't actually a node as well as if it isn't an instance and that made the crash bug related to passable in scr_movement_range_ai go away.  I detected in my debugging two other types that could end up in neighbor rarely, and those are the cause of this type of crash.  One type was obj_cursor_follower, and the other was not actually a single object type but it was some object that inherits from of obj_actor like obj_enemy_delian_ranger or obj_enemy_delian_knight.  I have tried to figure out where these get introduced in the code without any success.  The obj_cursor_follower object in particular befuddles me because there's not even any script I could locate that uses obj_cursor_follower except in obj_cursor_Create_0, the create event for obj_cursor, but it creates an instance of cursor_follower and doesn't even assign it to any variable that can be used to reference it.  I couldn't find any place in the code that uses the obj_cursor_follower type such as with the statement with(obj_cursor_follower) { } to iterate over objects of that instance type without the use of a stored variable of that type, so I don't have a clue how an obj_cursor_follower could ever end up assigned to the list of neighbors that neighbor iterates over.

Because I couldn't figure out where obj_cursor_follower somehow gets assigned to neighbor, I couldn't really fix the original root cause of the passable bug but I did at least find a way to prevent it from crashing the game by putting debug assertions in that force it to ignore any neighbor or current objects that aren't actually obj_node in scr_movement_range_ai.  After doing that I found through further testing that this bug also definitively occurs in scr_movement_range and scr_movement_range_hover as well which I fixed in the same way as movement_range_ai.  Using a grep tool on the script code I also found that passable is checked in an identical fashion in three other scr_movement_range functions: scr_movement_range_zone, scr_movement_range_ai_dragon, and scr_movement_range_hover_dragon, so I also patched those in the same way as the first 3 in order to prevent the same bug from potentially crashing them.

All total the mitigations for this bug involved implementing a new function
scr_debug_assert_instance_type
and placing 2 debug assertions (one for current and one for neighbor) in each of the 6 scr_movement_range files:
scr_movement_range
scr_movement_range_ai
scr_movement_range_ai_dragon
scr_movement_range_hover
scr_movement_range_hover_dragon
scr_movement_range_zone


Crash type 2 - Data structure with index does not exist

Example crash dump:
___________________________________________
############################################################################################
ERROR in
action number 1
of Step Event0
for object obj_cursor:

Data structure with index does not exist.
at gml_Script_scr_movement_range_hover
############################################################################################
gml_Script_scr_movement_range_hover (line -1)
gml_Object_obj_cursor_Step_0

Its a bit difficult to locate the exact code in scr_movement_range that causes an error like this because you can see it doesn't provide the name of the data structure with the index that does not exist.  It has to be some kind of structure that is normally accessed by index such as a ds_list or a ds_grid, but there are quite a few places where the code in scr_movement_range_hover reads values from ds_list structures via the ds_list_find_value() so to pinpoint the location of the bug is guesswork without some way to narrow it down.  To do that I created three new functions scr_set_attackdisplay_range1(node), scr_set_attackdisplay_range2(node), scr_set_attackdisplay_range3(node) because in the code there are three distinct sections where it performs a lot of ds_list_find_value operations to locate nodes and set the attackdisplay variable on them to true to paint the nodes at a certain distance away from the current node red on screen to indicate its possible to attack when the game is displaying a movement range visually on screen.  I moved the code from each section into one of those functions so that when the crash occurred I could figure out if it was one of those sections in particular causing the error. After a good bit of testing in this new code configuration all of the crashes I detected as far as I can remember were caused by the range3 section, which only applies to sniper characters.  This explains why many players have reported that not using sniper characters helped them get through chapter 22.  Every access of an index in this section is related to ds_list_find_value on either viewneighborsranged or viewneighborsmelee.  Examining the code in this section I found that it iterates over array index 0 through 3 of current.viewneighborsmelee then for each of those it iterates over array index 0 through 7 of the viewneighborsranged of each of those nodes.  Since the error says that the index does not exist as reported in the crash dump, it appeared that either viewneighborsmelee had less than 4 entries in its list or viewneighborsranged had less than 8 items in its list once in awhile so trying to access them crashed the game.

To fix that potential problem I rewrote the code in scr_set_attackdisplay_range3 so that instead of accessing a fixed number of indices with hardcoded values it would use a for loop and iterate only over whatever number of entries actually exist in viewneighborsmelee and viewneighborsranged using a for loop with ds_list_size() as the maximum limit and it seemed like the crashes had gone away.  I did some more grep searching to find all the code in the game that accesses viewneighborsranged or viewneighborsmelee with a fixed number of entries instead of iterating over ds_list_size() to try to resolve any other potential crashes related to the same issue, even though I did not actually experience any in my limited testing.  This led to me rewriting scr_set_attacksiplay_range1() and scr_set_attackdisplay_range2() with for loops and the debug assertion to limit operation to only on obj_node instances the same way as I had scr_set_attackdisplay_range3() as a precautionary measure although I don't know if that was necessary because I don't remember experiencing a crash in those. I also found code related to healers in obj_ai_controller_Step_0 and obj_purpleai_controller_Step_0 that operated on a fixed number of nodes of viewneighborsranged and viewneighborsmelee so I rewrote those as for loops that respect the actual size of the lists as well.

At this point I thought the changes to the way the game iterates over viewneighborsranged and viewneighbrsmelee would stop all the crashes, but after some more testing playing through chapter 22 I found that once in awhile I would still receive a crash in this section related to an index that does not exist, when clearly I was checking to make sure the indexes accessed existed before accessing them.  This had me stumped but due to my experience with the wrong type of node being received in the code related to the neighbors list while resolving crash type #1 and viewneigborsranged and viewneighborsmelee being derived ultimately from neighbors (in obj_game_Create_0) I decided to also put the scr_debug_assert_instance_type check right at the start of scr_set_attacksiplay_range3 to filter out anything that wasn't an instance of type obj_node. This stopped more crashes and I thought I had gotten them all so I released the first patch with it this way, but I ended up having to add more checks in the second patch when I later found that there were still crashes here in scr_set_attackdisplay_range3.


Other than code changes that are directly related to avoiding crash type #1 and crash type #2 in chapter 22, I implemented in the first patch two other changes that might not have been strictly necessary but which address other minor bugs or inconsistencies I came across.  I had thought that the crashes were due to the number of viewneighborsranged items.  While looking for instances of it being used with fixed indexes without testing the list size I discovered that there are two places that viewneighborsranged are set in objects.  One of those is in scr_obj_game_Create_0 and the other is in scr_obj_game_cutscene_Create_0.  In obj_game at the end of setting up the viewneighborsranged list and viewneighborsmelee list on a node it checked to see if the number of nodes was less than 8 for viewneighborsranged or less than 4 for viewneighborsmelee and if it was then it added 8 or 4 copies of the last node it had added, as a measure to try to ensure that there were at least 8 or 4 items in the list to try to resolve crash bug #2.  This code should actually create some viewneighborsmelee lists of nodes at the edge of the map with 7 nodes instead of 3 in them and some viewneighborsranged lists with 13-15 entries in them instead of 5-7.  I'm guessing this code must have been put in by someone trying to resolve the invalid index problem that caused crash #2, but since I made all the accesses of viewneighborsranged and viewneighborsmelee indices in the game iterate over the actual size of the list instead of checking a hardcoded amount of exactly 4 or 8 values, there's no reason to need to add excess values to those lists as a clunky workaround to prevent checking a non-existing node.  I wasn't sure if those excess entries that were added might be causing problems so I removed them as a preventive measure. I've looked at all the code in the game related to viewneighborsranged and viewneighbrosmelee and I see no need for them. obj_game_cutscene does not add excess entries like that.  Ultimately I think all the changes I made to make sure that the game wasn't accessing illegal indexes probably were not harmful but may be unneeded as it turned out that it was just that the game was trying to access the viewneighborsranged or viewneighborsmelee list on object types that don't even have them in scr_set_attackdisplay_range3, the spurious obj_cursor_follower and obj_actor descendants that I blocked with the debug assertions.  When trying to access a list on a variable that doesn't exist the crash dump was complaining about an invalid index instead of complaining about accessing a variable that doesn't even exist.

The last change I made was as a result of noticing while I was playing through chapter 22 dozens of times that reinforcements that spawn every other round on the right end of the map could stack up.  Two of the enemies run towards you after spawning no matter how close to you they are and the other two stay where they spawn if you don't get close enough to them.  If you wait a long time to approach the ones that don't move you can be surprised when suddenly you find out when they start moving that there are three or four enemies stacked there, not just one.  I stumbled upon the code that causes this monster spawning in obj_turn_scripter_22_Other_10 which I guess is the event that decides every turn if any enemies should spawn while I was looking for something else.  I found that the game actually is meant to check and make sure they don't spawn if the node they are supposed to spawn on still has an enemy in it, but the code to spawn the unit instance uses x and y values which are one pixel each and the code that checks to see if the node is already occupied uses GridX, GridY.  GridX is suposed to be equivalent to 32 * GridX and y to 32 * GridY but at some point someone changed one of the two there without realizing they needed to change the other or they did some faulty mental math with the result that the code was checking one node to make sure it was empty before spawing the enemy unit on a completely different node.  Clearly the code written there was originally intended to avoid spawning a second, third, or fourth unit on a node that was already occupied, so I in obj_turn_scripter_22_Other_10 I made code modifications to correct all four node test values so they are checking the same nodes as the ones that the four units are spawning on.  If that was not the original intention of the code there would have never been a check to see if the nodes were empty first at all and there would have just been the instance spawning code without any empty node test conditional first.


Version 1.0.1

It turned out that there was still crashes in scr_set_attackdisplay_range3 after version 1.0.0.  I'm not sure if the value of current going into scr_set_attackdisplay_range3 is ever actually invalid as I had thought, but I found during further testing that when node iterates over current.viewneighborsmelee and innernode iterates over node.viewneighborsranged, either node or innernode can end up being an obj_cursor_follower or obj_actor descendant occasionally, just like was happening in bug #1, so I instituted debug assertions on those types as well in scr_set_attackdisplay_range3.  That fixed the remaining crashes that I had detected so I believe there are no more crash bugs remaining in Chapter 22 with this patch.  Ultimately I never did figure out exactly how obj_cursor_follower or obj_actor descendants get into the neighbors, viewneighborsranged, or viewneighborsmelee, but when those invalid types are detected the code will just ignore them.


Version 2.0.0

In this version I set out to fix some movement related bugs in the game that I discovered awhile back that for sure occur with the movement speed set to movement speed 8 in the game settings.  To see one easily reproduced example you can move a character exactly 7 squares to the right, then have another character try to push him upwards. It will glitch out visibly. These bugs do not crash the game, but they cause various glitches.

It turns out that if you move a character exactly 7 squares on movement speed 8 then some kind of glitch occurs where the actor does not completely stop moving after it completes its movement path.  Instead of having an x or y value that is a whole number and a multiple of 32, it will have some value after the decimal as a result of drifting very slowly.  If you wait long enough it will drift into another tile which will produce problems like not being able to select the character by clicking on the tile that its sprite appears on, but being able to select it by clicking on another tile which doesn't have its sprite.  This can also result in more than one unit being on the same tile which should not happen.  The x or y value having a decimal portion will make it so that the position of the actor doesn't match the x and y of any obj_node and that will cause malfunction in code that tries to verify that or tries to compare the exact x and y position of the actor with the nodes.  If you try to chain an enemy that is drifting the character will never be able to move again when the chain is removed because it only restores the move speed of the character if it matches the stored position when the chain was created.  If you try to push a character that is not on x,y that have a multiples of 32, you'll see a blurry image where you pushed it from as it is sliding the image until it reaches the exact x,y of the character at which point it will remove the obj_push instance.  That never happens if the actor is aligned on an x,y that are multiples of 4, which it isn't if the character is drifting.

To work around the several different glitches caused by this drifting problem, I had found in the past that just increasing the movement speed in the options to 10 fixed the specific problem with units moving 7 tiles and didn't appear to create any other new problems, but still since the game defaults to movement speed 8 I believe most players who are unaware will eventually experience these bugs in the game.  I hypothesize that GameMaker should have automatically handled fully stopping the actor but isn't due to some floating point code that adds or subtracts floating point values and expects them to reach some exact amount which does not happen due to normal rounding errors associated with using floating point approximations of real fractions which can have an infinite number of digits after the decimal.  That would happen with some fractions and not with some other fractions so that might explain why it happens at one movement speed but not at another.  I could be totally wrong on that, but it is my assumption.  In any case the solution has to be to manually set the speed of every actor to zero and make sure its x and y are integral values with no digits after the decimal point after it finishes a movement path. 

The event for finishing a movement path on an actor is obj_actor_Other_8.  This event already exists, so I just modified the code for this event to add lines to set speed, hspeed, and vspeed to 0 in addition to what it was already doing and made sure the grid values it is using to set x and y are whole numbers. I implemented this in patch version 2.0.0 and although I didn't test every bug I previously found related to movement speed 8 individually I did spot check to make sure that if I test my easily reproducible case of moving a unit 7 squares to the right, then having another character push it, that does now work properly on movement speed 8.  This means that this fix has resolved the drift that was occurring and it should prevent all the related bugs which only occur when there is drift including any previously undiscovered drifting problem that could possibly occur when completing a path with any other arbitrary combination of movement speed and number of squares moved.  A fix like this wouldn't work if there were objects in the game that were intended to keep moving when not on a movement path or which used physics, like in an action arcade game, since it would stop objects that should be moving, but since there are not supposed to be any constantly moving actors in this game except those being driven around by a movement path it should work fine here.
